# RestX 开发宗旨

本文件适用于整个 RestX 仓库。所有开发者和代码代理在设计、实现、重构、测试及评审代码时，都必须遵守以下原则。

## 1. 扩展性优先

RestX 会持续增加彼此差异较大的功能。新增代码必须符合当前的特性模块架构，不能为了快速交付而重新形成集中式、强耦合实现。

- 一个业务功能应归属于独立的 `src/features/<feature-id>/` 特性胶囊。
- 特性可以分别拥有 `renderer`、`main`、`preload`、`shared`、样式、状态和测试。
- 左侧菜单与页面路由由 renderer 特性声明统一生成。新增菜单不得在 Shell 或 Router 中加入特性专用判断。
- 需要系统权限的功能通过该特性自己的 main handler 和 preload 白名单 API 暴露，不得向 renderer 提供任意 IPC channel 调用能力。
- 特性的业务状态、样式、存储和异常处理应由该特性自行管理。单个特性发生错误时，不得影响其他菜单和应用外壳。
- 特性不得直接引用另一个特性的内部页面、组件、状态或服务。确需协作时，应通过公开契约或 capability，并显式声明依赖。
- `src/platform/` 只保留应用生命周期、特性注册、依赖校验、安全边界、错误隔离等架构能力，以及确定会被大多数特性依赖的稳定通用代码。
- 只被一个或少数特性使用的业务实现不得提前放入 platform 或创建万能 shared 模块。
- 新增或删除特性后，必须验证重复 id、重复路由、IPC channel、依赖缺失、依赖循环和跨特性私有导入。

详细目录与接入方式见 `docs/feature-architecture.md`。

## 2. 蓝区与黄区

RestX 的代码按开放范围分为蓝区和黄区。这是代码归属和信息边界，不代表两套不同的架构。

### 蓝区（开放区）

蓝区是默认区域，包含可公开、可复用、可审查的代码：

- 平台框架、特性注册机制和生命周期管理；
- 通用类型、接口、数据契约和 capability 定义；
- 通用 UI、基础组件、安全封装和基础工具；
- 不包含专有实现的业务流程骨架；
- 使用公开或合成数据的测试、示例和开发文档。

除非用户明确指定为黄区，新代码默认归入蓝区。

### 黄区（保密区）

黄区只承载少量必须保密的专有实现，例如特定算法、策略或实现性质的函数体。

- 黄区必须保持最小化，不得把完整特性、框架代码或通用能力整体放入黄区。
- 接口、类型和通用契约应定义在蓝区；黄区只实现这些契约。
- 黄区代码可以依赖蓝区契约，蓝区业务代码不得直接依赖或了解黄区实现细节。
- 黄区能力应通过依赖注入、窄接口或受控适配器接入。组合入口可以完成装配，但不得重新导出黄区内部实现。
- 不得将黄区源码、专有规则、真实样本或可推导实现细节的内容复制到蓝区。
- 不得把黄区内容写入日志、错误消息、遥测、测试快照、公开测试数据、注释、README、OpenSpec 或模型提示词。
- 黄区测试应留在相同保密边界内；蓝区只针对公开契约使用合成数据做黑盒验证。
- 已被标记为黄区的代码，不得在未经用户明确同意的情况下移动、改写为开放实现或扩大可见范围。

## 3. 边界判断

开始编码前，应先确认代码属于哪个特性、哪个进程以及哪个开放区域。

1. 先确定功能所有者，避免把业务代码放入 platform。
2. 再确定 renderer、main、preload 或 shared 边界。
3. 最后判断蓝区或黄区；无法确定时停止扩散，向用户确认。
4. 若一个文件同时包含通用框架与专有实现，应拆分文件：契约和框架留在蓝区，最小实现留在黄区。

保密边界不得以牺牲类型安全、测试能力或模块隔离为代价。应通过接口和装配解耦，而不是复制代码、使用全局变量或绕过现有注册机制。

## 4. 开发与验证要求

- 保持改动聚焦，避免顺手修改无关特性。
- 保持现有公开 API、用户数据格式和持久化路径兼容；需要破坏性变更时必须先说明迁移方案。
- 所有 IPC 输入都要校验，channel 必须使用所属特性的命名空间。
- 日志不得包含凭据、配置原文、用户隐私或黄区实现细节。
- 新增架构能力需要相应的边界测试；修改业务功能需要相应的行为或回归测试。
- 提交前至少运行 `pnpm test`、`pnpm typecheck` 和 `pnpm build`，并确保 `git diff --check` 通过。
- 不得提交与当前任务无关的本地文件、密钥、日志、构建产物或用户草稿。

## 5. 独立验证任务与推送门禁

功能开发、UI 调整或 Electron 行为修改完成后，主开发任务不得立即推送。应先形成一个稳定的本地代码检查点，再根据实际变更风险选择适用的验证任务，并把彼此独立的验证工作分发到独立的 Codex 后台任务；不得默认执行全部验证。用户可以同时在主工作区进行人工验证。

### 验证任务划分

- 自动化验证任务：在独立 worktree 中运行 `pnpm typecheck`、`pnpm test`、`pnpm build` 和 `git diff --check`。
- 视觉验收任务仅在任务与 UI 强相关时适用，例如界面布局、样式、可见状态、关键 UI 交互、窗口尺寸或操作系统级显示发生变化。在隔离的 renderer 环境中检查需求涉及的固定页面状态、关键交互、窗口尺寸和截图，不得在验收过程中修改样式或业务代码。
- 纯业务逻辑、数据处理、依赖兼容、IPC、后端或主进程修复，如果没有改变用户可见界面，不得启动视觉验收。是否需要截图应以验收标准能否通过视觉证据验证为判断依据；不适用时标记为 `N/A`，不得记为 `FAIL` 或阻塞推送。
- 进程冒烟任务：验证应用启动、主窗口加载、主进程存活、启动日志和正常退出，默认时间盒为 5 分钟。
- 视觉验收默认时间盒为 8 分钟。超时、环境异常或发现新范围时，应返回阻塞原因，不得自行扩大到打包、签名、公证或安装包验证。
- 打包和操作系统级显示行为属于独立验收任务，只有用户明确要求时才能执行。

### 隔离与权限

- 验证任务只读使用待验收代码，不得修改业务文件、提交、合并或推送。
- 主任务是唯一负责修复、整合和推送的任务。验证失败后，由主任务修复，再只重跑受影响的验证任务。
- 自动化验证和视觉验收应优先使用独立 worktree、独立端口和临时输出目录，避免影响用户正在运行的应用。
- 进程冒烟不得与用户正在使用的 RestX 实例争用 `~/.restx/runtime`。没有测试专用数据目录时，应等待人工验证实例退出后再执行。
- 验证任务必须返回结构化结果：检查范围、执行命令或步骤、通过项、失败项、证据、耗时以及最终 `PASS`/`FAIL`。

### 推送条件

只有适用的自动化验证、视觉验收和进程冒烟任务全部为 `PASS`，才允许推送。若用户表示要进行人工验收，还必须等待用户明确确认；若用户已明确授权“验证通过后直接推送”，则不重复询问。

无法创建独立后台任务时，主任务仍须按相同测试范围和推送门禁串行执行，并明确说明未能分发的原因。

## 6. 核心判断标准

一次改动只有同时满足以下条件，才符合 RestX 的开发宗旨：

- 新功能可以独立新增、禁用或删除；
- 不要求无关特性跟随修改；
- 公共层没有吸收特性专用逻辑；
- 蓝区只依赖公开契约，不泄露黄区实现；
- 安全边界、测试和现有用户行为仍然成立。
